## 学习内容

### 阅读：RAG（检索增强生成）

读了 [AI × Web3 School Handbook — RAG 章节](https://aiweb3.school/zh/handbook/ai/rag/)

### 关键概念 / 收获

1. **RAG 的本质** — 不是"给模型接一个向量库"，而是一条把外部知识取回、筛选、引用、交给模型使用的证据链。RAG 不是为了把回答变长，而是让回答有来源、有版本、有边界。

2. **缺少 citation 和 freshness 的 RAG** — 只是把幻觉从模型内部搬到了检索系统里。RAG 的可靠性取决于证据链，不取决于"向量库"这个名词。

3. **三个核心判断点**：
   - 文档怎么切（Chunking）
   - 查询时取哪些内容（Retrieval）
   - 生成时如何引用和拒答（Citation & Refusal）

4. **Chunking** — 按结构切（标题、API endpoint、函数说明、FAQ），每个 chunk 保留来源 URL、标题路径、更新时间和版本。技术文档要小心跨段落信息。

5. **Vector DB** — 不仅要存向量，还要存 metadata：来源、版本、chain、更新时间、可信等级、是否废弃。向量相似 ≠ 答案正确。

6. **Retriever** — 不能只看语义相似度，还要知道用户问的是哪个产品、版本、时间段。如果有版本/环境/时间/地址关键词，就不能只做纯向量搜索。

7. **Rerank** — 初筛后重新排序，把更可信的排前面。trade-off：增加延迟和成本，面向资产风险或开发者文档值得加。

8. **Citation 是 RAG 的灵魂** — 至少要说明：来自哪份文档、是否官方、版本/更新时间、哪些只是模型归纳、哪些地方没足够证据。

9. **RAG 在 AI x Web3 中的位置** — 位于 Knowledge Base 和模型之间，帮 Agent 查资料、补上下文、引用证据。但 RAG 结果不能直接触发链上动作，还要接 simulation、policy 和 human check。

### 与之前章节的联系

- **与 Context 章节**：RAG 本质上是在构建"外部知识层"的上下文。Context 章节提到的层级设计（指令→任务→事实→知识→记忆），RAG 负责的就是"知识层"那块。上一章说"模型看到了但没抓住重点"问题，RAG 用检索+rerank+citation 来解决。
- **与 Prompt 章节**：好的 RAG 输出结构需要好的 prompt 设计来约束——强制区分 answer / sources / uncertainties / needs_version_check。

### 最小实践

章节末尾的动手题：做一个"协议文档 RAG 问答"最小版本。选一个官方文档站，抓 10-20 篇页面，按标题结构切 chunk，存入向量库。用户提问时返回答案和引用。至少测试三类问题：明确存在的、不存在的（拒答）、旧版本的（提示核对版本）。

输出格式：
- answer
- sources
- uncertainties
- needs_version_check

### RAG 完整流程分析

RAG 整体分两条线：离线的索引管线 & 在线的查询管线。

#### 🔧 索引管线（Indexing Pipeline）— 离线

```
原始文档
    │
    ▼
┌─────────────┐
│ 文档收集     │  ← 确定数据源：官方文档、GitHub Wiki、PDF、Notion
│ & 清洗       │  ← 去重、去噪、标准化格式
└─────────────┘
    │
    ▼
┌─────────────┐
│ 结构解析     │  ← 识别标题层级、代码块、表格、列表
│             │  ← 保留页面 URL、更新时间、版本号
└─────────────┘
    │
    ▼
┌─────────────┐
│ Chunking    │  ← 按结构切（标题、API endpoint、函数说明、FAQ）
│ (分块)      │  ← 每个 chunk 附 metadata：来源 URL、标题路径、版本、时间、可信等级
└─────────────┘
    │
    ▼
┌─────────────┐
│ Embedding   │  ← 把每个 chunk 转为向量
│             │  ← 模型选择影响检索质量
└─────────────┘
    │
    ▼
┌─────────────┐
│ 存入         │  ← Vector DB + Metadata 索引
│ Vector DB   │  ← 同时支持向量检索和结构化过滤
└─────────────┘
```

#### 🔍 查询管线（Query Pipeline）— 在线

```

    ▼
┌─────────────┐
│ 查询理解     │  ← 意图分类、关键词提取、query 改写
│ & 改写       │  ← 比如 "approve风险" → "approve 函数 安全注意事项"
└─────────────┘
    │
    ▼
┌─────────────┐
│ 检索         │  ← 三种策略通常配合使用：
│ Retrieval   │
│  ┌────────┐ │    1. 向量检索（语义相似度）
│  │ Vector │ │    2. 关键词检索（BM25，精确匹配）
│  │ Search │ │    3. Metadata 过滤（版本、产品线、时间范围）
│  └────────┘ │
│  ┌────────┐ │
│  │Keyword │ │
│  │ Search │ │
│  └────────┘ │
│  ┌────────┐ │
│  │Metadata│ │
│  │ Filter │ │
│  └────────┘ │
└─────────────┘
    │
    ▼
┌─────────────┐
│ 合并 & 排序  │  ← 把向量检索、关键词检索的结果合并去重
│ Merge       │  ← 初步排序（分数融合）
└─────────────┘
    │
    ▼
┌─────────────┐
│ Rerank      │  ← 用专门的 reranker 模型重新打分排序
│ (重排序)     │  ← 把真正相关的排前面，滤掉"标题匹配但内容不对"的结果
│             │  ← 视角差异：向量检索只看 embedding，reranker 能看到完整文本
└─────────────┘
    │
    ▼
┌─────────────┐
│ Context     │  ← 把 top-K 个 chunk 组装成 context
│ Assembly    │  ← 排序：最相关的放前面；注入来源标记 [1][2][3]
│             │  ← 加入指令约束："只基于下面材料回答，不确定就说不知道"
└─────────────┘
    │
    ▼
┌─────────────┐
│ LLM         │  ← 模型基于 context 生成回答
│ Generation  │  ← 输出格式应包含：answer / sources / uncertainties / needs_version_check
└─────────────┘
    │
    ▼
┌─────────────┐
│ 后处理       │  ← 检查 citation 是否有效、是否拒答失败
│ Post-       │  ← 格式化展示给用户
│ processing  │
└─────────────┘
```

#### 🧠 关键设计决策

| 决策点 | 好坏对比 |
|--------|---------|
| Chunk 大小 | 太小（<100 tokens）→ 语义碎；太大（>1000）→ 噪声多 |
| 重叠窗口 | chunk 之间留 10-20% 重叠，防止关键句正好被切在边界 |
| 检索策略 | 纯向量 → 漏精确匹配；纯关键词 → 漏语义相关；混合 → 最优 |
| Metadata 过滤 | 没有 → 旧版本文档污染结果；有 → 精准定位 |
| Top-K 取值 | 取太少 → 可能缺关键证据；取太多 → token 爆炸、注意力稀释 |
| Rerank | 加 → 质量提升明显，但延迟增加 100-500ms |
| Citation 粒度 | 粗（"来源：文档"）→ 没法验证；细（段落+URL）→ 可信任 |

#### ⚠️ 常见翻车现场

1. **检索到但用不上** — 模型拿到材料但不引用，自己"自由发挥" → 加 strict prompt 约束
2. **相关但不正确** — 旧版本文档高度匹配但答案已过期 → 靠 metadata 版本过滤
3. **多语言冲突** — 英文问题命中中文文档，embedding 模型多语言能力要足够强
4. **拒答失败** — 没有相关材料但模型硬编 → 需在 prompt 层强制"材料不够就拒答"
5. **Citation 漂移** — 模型引用了一个 chunk，实际答案是从自己知识里编的 → 需要 citation recall 验证

#### 🔗 AI x Web3 场景的特殊延伸

在这个场景里，RAG 全流程还要多几个环节：

```
文档 → Chunk → Embed → 存 DB
                          │
用户提问 → 检索 → Rerank → Context → LLM → 输出带引用
                                               │
                                          ⚠️ 如果是链上动作意图
                                               │
                                          Simulation
                                               │
                                          Policy Check
                                               │
                                          Human Confirm
```

RAG 输出的内容不能直接驱动链上操作。文档说某函数"可以调用"，不等于当前用户"应该签名"——真正执行前还要模拟交易、检查策略、人工确认。
